# antifragile.md — 反脆弱设计

## 核心理念

系统不仅要能应对故障，还要能从故障中变得更强。每次故障都是一次学习机会。

## 数据源降级策略（三级降级）

```
Level 1: 主数据源（东方财富/同花顺）
  ↓ 失败
Level 2: 备用数据源（新浪/腾讯）
  ↓ 失败
Level 3: 标注"数据缺失"
```

### 降级触发条件

| 故障类型 | 触发条件 | 降级动作 |
|----------|----------|----------|
| API超时 | 响应时间>30秒 | 切换备用源 |
| API断连 | RemoteDisconnected | 切换备用源 |
| 数据异常 | 返回全0/空值 | 标注"数据缺失" |
| 限流 | ret=-2 / 429 | 等待后重试 |

### 降级恢复

- 备用源成功后，自动切回主数据源
- 每日首次采集时检测主数据源可用性
- 主数据源恢复后，清除降级标记

## API故障应对预案

### 东方财富API断连

**症状**：`ak.stock_zh_a_hist()` 返回 `RemoteDisconnected`

**预案**：
1. 立即降级至新浪 API
2. 使用 Ashare 双源实时行情
3. 记录故障时间，分析故障模式

### 同花顺API限流

**症状**：`ak.stock_board_industry_summary_ths()` 返回空数据

**预案**：
1. 等待60秒后重试
2. 使用东方财富 `ak.stock_sector_fund_flow_rank()` 做交叉验证
3. 标注【单位待核验】

### 新闻源故障

**症状**：news_YYYY-MM-DD.json 中某源为空

**预案**：
1. 其他源正常则继续
2. 全部源故障则标注"新闻数据缺失"
3. 使用 web_search 作为最后备用

## 模型切换策略

| 场景 | 主模型 | 备用模型 | 切换条件 |
|------|--------|----------|----------|
| 日常分析 | mimo-v2.5-pro | deepseek-v4-pro | mimo不可用 |
| 视觉任务 | mimo-v2-omni | - | 需要看图时 |
| 复杂推理 | deepseek-v4-pro | mimo-v2.5-pro | 推理质量不足 |

### 切换触发

- 主模型连续3次返回错误 → 自动切换备用模型
- 主模型响应时间>60秒 → 自动切换备用模型
- 手动切换：`/model deepseek-v4-pro`

## 推送失败重试机制

### 重试策略

| 重试次数 | 等待时间 | 动作 |
|----------|----------|------|
| 第1次 | 5秒 | 重试同一通道 |
| 第2次 | 15秒 | 重试同一通道 |
| 第3次 | 30秒 | 切换备用通道 |
| 第4次 | 60秒 | 切换备用通道 |
| 第5次 | - | 记录失败，通知主人 |

### 通道优先级

1. 微信（主通道）
2. API Server（备用通道）
3. 本地文件（最后备用）

### 重试触发

- 推送返回错误 → 自动重试
- 推送超时（>30秒）→ 自动重试
- 限流（ret=-2）→ 等待后重试

## 观察池自动退池机制

### 退池触发条件

| 条件 | 动作 | 时限 |
|------|------|------|
| 止损 | B池-5%, A池-8% | 立即 |
| 超时无触发 | C池>15天→退池, B池>30天→降级C | 每日检查 |
| 赶顶信号 | RSI>75+量比<0.8+位置>90% | 降级C |
| 基本面恶化 | ST/非标审计/质押>70% | 直接退池 |
| 板块退潮 | 连续3日资金净流出 | 标记观察 |
| 触发失效 | 失效条件达成 | 退池 |

### 退池记录

每次退池必须记录：
- 退池时间
- 退池原因
- 退池前状态
- 退池后处理

## Cron调度器可靠性

### 保底机制

1. 系统 crontab 三重重启（08:55/11:55/15:25）
2. 每次推送前检查 ticker 状态
3. ticker 停止超过30分钟 → 自动重启

### 故障恢复

```bash
# 检查 ticker 状态
grep "Cron ticker started" ~/.hermes/logs/agent.log | tail -1

# 手动重启
hermes gateway restart

# 验证恢复
cronjob action='list'  # 确认 next_run_at 在未来
```

## 系统健康检查

### 每日检查项

| 检查项 | 频率 | 异常处理 |
|--------|------|----------|
| Gateway状态 | 每小时 | 自动重启 |
| Cron ticker | 每次推送前 | 自动重启 |
| 数据源可用性 | 每日首次采集 | 降级备用源 |
| 模型可用性 | 每次调用 | 切换备用模型 |
| 磁盘空间 | 每日 | 清理旧文件 |

### 健康指标

| 指标 | 正常范围 | 异常处理 |
|------|----------|----------|
| API响应时间 | <30秒 | 降级备用源 |
| 推送成功率 | >95% | 检查通道 |
| 数据采集成功率 | >90% | 检查数据源 |
| 模型调用成功率 | >95% | 切换备用模型 |

## 从故障中学习

### 故障记录

每次故障后，记录到 `logs/update_log.md`：

```markdown
## YYYY-MM-DD HH:MM - 故障标题

**类型**：系统故障
**影响**：影响范围
**根因**：故障原因
**修复**：修复措施
**预防**：预防措施
```

### 故障分析

每月分析故障模式：
- 高频故障源 → 优先优化
- 故障恢复时间 → 缩短恢复时间
- 故障影响范围 → 缩小影响范围

### 知识沉淀

将故障处理经验沉淀到：
- `eval/failure_rules.md`：故障排查手册
- `rules/data_rules.md`：数据源降级策略
- `advanced/antifragile.md`：反脆弱机制
