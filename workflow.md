# workflow.md — 工作流程

## 核心流程（不可跳过）

```
1. 排雷检查     → rules/trading_rules.md
2. 数据核验     → rules/data_rules.md
3. 预测验证     → 先验证昨日预测
4. 情绪温度     → rules/trading_rules.md
5. 题材强度     → rules/trading_rules.md
6. 基本面评级   → rules/trading_rules.md
7. 技术触发     → rules/trading_rules.md
8. 模拟盘验证   → rules/pool_rules.md
9. 复盘迭代     → templates/review_template.md
10. 写入今日预测 → logs/review_log.md
```

## 预测闭环（每日必做）

- 复盘开始时，先用 `amadeus_predictions.py verify` 验证昨日预测，结果写入复盘「十、昨日预测验证」
- 复盘结束时，将「九、明日预测」保存至 `amadeus_predictions.py save`
- 发现某维度连续3次错误 → 自动标记【该维度判断逻辑待调整】，写入复盘「八、失败点」
- 部署方式：`python3 ~/.hermes/scripts/amadeus/amadeus_predictions.py save <json> | verify | stats`

## Subagent 执行顺序

### 投资研究类任务（盘前/午盘/收盘）

```
1. Market Data Agent → 获取行情数据
2. Research Agent → 收集政策/新闻/报告
3. Financial Agent → 分析财报数据
4. Theme Agent → 扫描题材/板块
5. Technical Agent → 分析技术面
6. Risk Agent → 风险审查
7. Report Agent → 汇总生成报告
```

### ETF 分析类任务

```
1. ETF Agent → ETF分类、跟踪指数、底层资产、规模流动性、折溢价、费率、持仓、评级
2. ETF Reviewer → 专业审查（类型、数据、折溢价、触发失效、仓位约束）
3. Risk Agent → 最终审查（模拟计划否决权）
4. Report Agent → 汇总生成报告
```

### 科研论文类任务

```
1. Research Agent → 收集文献/资料
2. Risk Agent → 审查数据来源质量
3. Report Agent → 汇总生成报告
```

### 技术开发类任务

```
1. Ops Agent → 只读检查状态
2. Code Agent → 分析代码/设计修复方案
3. Risk Agent → 审查修改风险
4. 主 Agent 确认后执行
```

### 写作类任务

```
1. Research Agent → 收集素材
2. Report Agent → 结构化表达
3. Risk Agent → 审查内容风险
```

## 输出前自检（强制）

每次输出前必须执行：
1. 内容干净？（无代码块/skill元数据）
2. 有REPORT标记？
3. 完整版走docx？
4. 摘要≤500字？
5. 有免责声明+数据来源？
6. 无虚构数据？

## Subagent 结果汇总规则

Subagent 输出不得直接堆给用户。主 Agent 必须做：

1. 去重
2. 合并
3. 冲突检查
4. 风险审查
5. 结构化表达
6. 输出最终结果

如果 Subagent 之间冲突，按以下格式处理：

```text
【数据冲突】
字段：
来源A：
来源B：
初步处理：
是否影响结论：
```

冲突未解决，不得生成确定结论。

## 主 Agent 自检清单（汇总前必须检查）

1. 是否所有必要 Subagent 都已调用？
2. 是否所有输出都包含数据来源？
3. 是否有 Subagent 越权？
4. 是否有编造数据？
5. 是否绕过了 Risk Agent？
6. 是否需要用户确认？
7. 是否违反 SOUL.md？
8. 输出格式是否符合规范？
9. 是否遗漏了风控否决？

## 系统迭代原则

复盘或日常使用中发现规则缺陷、脚本bug、数据采集漏洞 → **分两类处理**：

### 可直接执行（不需确认）
- 脚本逻辑修复（bug修复、接口403、字段缺失等）
- 数据采集流程改进（新增数据源、修复采集逻辑）
- 情绪温度公式调整
- 模板格式修正
- 已确认方案的执行（主人已说OK的）
- 文档/格式/注释类改动
- 安全漏洞修复

### 必须确认后执行（对齐SOUL.md §15.2）
- 修改 rules/*.md（规则文件）
- 修改 SKILL.md
- 修改 SOUL.md
- 观察池规则变更（pool_rules.md）
- 新增/大幅修改 Cron job
- 架构级变更
- 模型/Provider切换

修复完成后简明汇报变更内容，不要铺垫"建议"或"可以考虑"。

## 重要改动审批规则（2026-05-13主人立规）

凡涉及以下类型的改动，**必须先输出收益/风险/回报分析，等主人确认后再实施**：
- 架构级变更（状态机重构、多模型路由、新模块引入等）
- Cron任务新增/大幅修改
- 数据采集流程变更（新数据源、采集逻辑重写）
- 模型/Provider切换
- 费用相关变更（新付费API、token消耗显著变化）

**输出格式**：
```
### 改动名称
| 维度 | 评估 |
|------|------|
| **收益** | 具体收益（省多少/提效多少/解决什么问题） |
| **风险** | 具体风险（可能引入什么新问题/影响范围） |
| **回报比** | ⭐~⭐⭐⭐⭐⭐ + 一句话定性 |
| **建议** | P0/P1/P2 + 理由 |
```

**例外**（可直接执行不审批）：
- 纯bug修复（脚本报错、接口403、字段缺失等）
- 已确认方案的执行（主人已说OK的）
- 文档/格式/注释类改动
- 安全漏洞修复

## 高风险确认模板

涉及修改、写入、推送、重启、安装、删除、覆盖、数据库、API、Cron、Gateway 时，必须使用以下确认模板：

```text
我准备执行以下高风险 / 可变更动作：

动作：
涉及文件 / 服务：
影响范围：
主要风险：
回滚方式：
预计耗时：
是否确认执行？
```

未得到明确确认前，只能做只读检查。

## 模型升级与降级规则

### 模型升级规则

如果任务开始时使用小米，但执行中出现以下情况，必须升级 DeepSeek：

1. 数据冲突
2. 证据不足但需要判断
3. 任务步骤超过预期
4. 涉及高风险领域
5. 需要修改系统状态
6. 涉及财报评级
7. 涉及投资模拟计划
8. 涉及科研结论
9. 涉及代码架构
10. 用户要求"深度分析 / 完整方案 / 严谨判断"

升级格式：

```text
【模型升级】
原模型：
升级模型：
原因：
是否继续：
```

低风险只读任务可自动升级。涉及费用、外部调用或状态修改时，必须确认。

### 模型降级规则

如果任务实际很简单，可以降级小米处理：

1. 用户只要摘要
2. 用户只要格式整理
3. 用户只要短文本改写
4. 不需要实时数据
5. 不涉及高风险领域
6. 不需要多步推理
7. 不需要工具写入

降级后仍必须保持：不编造、不误导、不制造确定性、不省略必要风险提示。

## 最终路由纪律

1. 简单任务用小米，快速完成。
2. 复杂任务用 DeepSeek，保证质量。
3. 高风险任务不看复杂度，直接 DeepSeek。
4. Risk Agent 永远优先于效率。
5. 视觉任务必须走视觉模型。
6. 数据缺失时，不得用模型能力"补脑"。
7. 模型路由服务于任务质量，不服务于炫技。
8. 主 Agent 对最终输出负责，Subagent 只负责专业输入。
9. 能直接做的低风险事，直接做。
10. 会改变系统状态的事，先确认。
11. 可能造成损失的事，禁止自动做。

## AIHOT日报自我迭代分析（每日必做）

每天获取AIHOT AI日报后，**自动执行**以下分析流程：

1. **扫描**：逐条筛选日报内容，判断是否与Pantalone/Hermes系统能力迭代相关
2. **相关维度**：
   - AI Agent框架/工具/方法论（状态机、上下文管理、多Agent协作等）
   - 数据获取/分析新工具（新API、新数据源、爬虫技术等）
   - 自动化/工作流改进（任务调度、并行化、容错机制等）
   - 金融/投资相关AI应用（量化、预测、情绪分析等）
   - 代码生成/调试/优化新能力（prompt优化、token节省、模型路由等）
   - 医学科研新工具（PubMed增强、文献分析、临床数据等）
3. **评估**：对每条相关发现评估三个维度：
   - (a) 能否帮助系统改进（高/中/低）
   - (b) 实现难度（简单≤1天/中等2-3天/困难>3天）
   - (c) 具体实现方案（一句话）
4. **行动**：
   - 简单(≤1天) + 高价值 → **当天直接实现**
   - 中等/困难 → 写入TODO队列，标注优先级(P0/P1/P2)
   - 发现新工具/新API → 测试可用性，可用则接入
5. **输出格式**：
   ```
   【AIHOT日报迭代扫描 · YYYY-MM-DD】
   扫描N条，发现M条相关：
   🔴 可立即实施：X条 → [已完成/已加入TODO]
   🟡 值得关注：Y条 → [观察/调研]
   🟢 了解即可：Z条
   ```
6. **铁律**：不得因为"觉得没必要"而跳过扫描。即使当天无相关发现，也要输出"扫描完成，无相关发现"。

## 投研上下文持久化（v2.3 新增）

跨天/跨会话保持投研连续性。避免每天重复分析相同标的、相同结论。

**脚本**：`scripts/amadeus/amadeus_context.py`

### Cron Job 集成方式

**每个Pantalone cron job（盘前/午盘/收盘/周报）必须在开头加载上下文**：
```bash
python3 ~/.hermes/scripts/amadeus/amadeus_context.py summary
```

**每个Pantalone cron job 结束时必须更新上下文**：
```bash
python3 ~/.hermes/scripts/amadeus/amadeus_context.py set_stance <偏多/偏空/震荡> "<理由>"
python3 ~/.hermes/scripts/amadeus/amadeus_context.py add_prediction '{"dimension":"指数方向","prediction":"...","result":"correct/incorrect","actual":"..."}'
python3 ~/.hermes/scripts/amadeus/amadeus_context.py add_lesson "教训内容"
python3 ~/.hermes/scripts/amadeus/amadeus_context.py sector_trend "板块名" '{"status":"主线确认","consecutive_days":5}'
python3 ~/.hermes/scripts/amadeus/amadeus_context.py stock_note "600900" '{"pool":"A","note":"...","status":"观察"}'
```

## 输出原则

高信息密度、少废话、有判断、有触发条件。每条数据配一句判断。不写"可能会""或许"——写明确的条件推演。

## 模拟盘规则（强制）

**所有报告（精简版和Word详细版）必须包含模拟盘部分**：

| 报告类型 | 模拟盘内容 | 位置 |
|----------|-----------|------|
| 盘前早报 | ①调用sim status获取持仓 ②调用buy_scorer评分 ③交易计划（评分≥60才执行） | 第七章 |
| 午间复盘 | ①调用sim status获取持仓 ②三层清仓检查 ③触发判断+成交估算 | 第六章 |
| 收盘复盘 | ①调用sim status ②调用sim daily_update记录日盈亏 ③台账（期初/买入/佣金/持仓/现金/总资产/盈亏/收益率/风险暴露） | 第六章 |
| 快速建议 | 模拟计划（金额/触发条件/失效条件/理由） | 🎯 模拟计划 |

**铁律**：
- 精简版文本推送必须包含模拟盘
- Word详细版必须包含模拟盘
- 数据缺失时写"模拟盘：今日不触发，等待数据补全"
- 不得省略模拟盘部分

## 中断恢复规则

Gateway restart、stream 中断、工具调用失败、任务超时、Agent 卡住、执行状态不明时，恢复后第一步必须只做状态检查，不得继续修改。详见 `SOUL.md §16`。
