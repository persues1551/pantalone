# 观察池验证迭代系统

## 核心流程
入池 → 1天观察 → 验证 → 迭代逻辑 → 复利积累

## 验证规则
| 涨跌幅 | 结果 | 处理 |
|--------|------|------|
| 涨幅>2% | success | 升级池子（C→B→A） |
| 跌幅>3% | fail | 退池 |
| 其他 | partial | 延长验证1天 |

## 验证脚本
pool_verify.py 支持：
- list：查看待验证股票
- verify CODE RESULT EVIDENCE LESSON：单只验证
- batch JSON_FILE：批量验证
- report：生成验证报告
- iterate：迭代分析（成功/失败模式）

## 迭代分析维度
1. 逻辑类型成功率：哪种逻辑胜率最高？
2. 失败原因分析：估值过高？业绩不及预期？行业转冷？
3. 优化建议：增加过滤条件、调整权重

## 定时任务
- 每天18:00：验证检查（job: e44aa2dd5c8a）
- 每天16:00：验证执行（job: 2361a68de7e2）

## Pitfall
1. 验证期太长会错过短线机会：当前1天验证期，快速验证快速决策
2. 验证结果需人工确认：自动涨跌幅判断只是初步，需结合新闻/资金/技术面
3. 迭代不能只看胜率：盈亏比也很重要，胜率60%但亏损大的策略不如胜率50%但盈亏比好的
4. 退池后不能立即重新入池：需等待新的逻辑假设，避免"报复性入池"
