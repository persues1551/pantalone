# 最小可行拆分模式（Minimum Viable Split）

## 适用场景

当一个核心配置文件（SOUL.md、SKILL.md等）膨胀到500+行、混合了多种职责时，需要拆分但不能破坏已有引用关系。

## 执行步骤

### 1. 只读审计（不修改文件）

```bash
# 统计总行数
wc -l <file>

# 找出所有章节边界
grep -n '^## ' <file>
grep -n '^### ' <file>

# 统计各章节行数
awk '/^## /{if(start)print start"-"NR-1" ("NR-start" lines)"; start=NR} END{print start"-"NR" ("NR-start+1" lines)"}' <file>
```

### 2. 分类每个章节

| 类别 | 动作 | 目标文件 |
|------|------|----------|
| 最高原则/身份/边界 | 保留在原文件 | SOUL.md |
| 路由/触发/模型选择 | 移出 | router.md |
| 执行流程/汇总/审批 | 移出 | workflow.md |
| 角色权限/能力边界 | 移出 | subagents/*.md |
| 领域规则 | 移出 | rules/*.md |

### 3. 输出确认模板

在修改前，必须输出变更确认模板，包含：
- 每个文件的变更类型（新建/重写/合并）
- 行数变化
- 影响范围
- 回滚方式

等用户确认后再执行。

### 4. 执行拆分

**顺序**：新建目录/文件 → 重写原文件 → 合并已有文件

**关键规则**：
- 原文件中移出的章节，只保留一行指向新文件的引用
- 新文件之间不要互相引用形成循环
- 每个文件开头标注"参见"链接

### 5. 验证

```bash
# 统计新文件行数
find <dir> -name '*.md' -exec wc -l {} +

# 检查引用完整性
grep -r 'SOUL.md §' <dir>  # 不应有旧的§号引用
grep -r 'router.md' <dir>  # 应该有新的引用
```

## 拆分目标形状

```
核心文件（SOUL.md）→ 路由（router.md）→ 流程（workflow.md）
                                            ↓
                                    角色定义（subagents/）
                                            ↓
                                    领域规则（rules/）
                                            ↓
                                    输出模板（templates/）
```

## 注意事项

- 拆分前必须创建备份（`<file>.bak.YYYYMMDD`）
- 拆分是"只读审计+确认+执行"三步，不能跳过审计和确认
- 拆分后总行数不变（只是重新组织），不会丢失内容
- cron prompt 不应直接引用 § 号，应引用文件名

## ⚠️ 坑：拆分后遗留 § 引用（2026-05-14 发现）

拆分后最常见的问题是**旧的 § 号引用失效**。例如：
- workflow.md 引用 `SOUL.md §5.4` 但 SOUL.md 已无 §5.4（重编号为 §15.4）
- risk_rules.md 引用 `SOUL.md §18.4` 但 SOUL.md 已无 §18.4

**拆分后必须执行**：
```bash
# 检查所有文件中的旧§引用
grep -rn "SOUL.md §" <dir>/{router,workflow,rules/*,subagents/*}.md
# 对每个找到的引用，验证目标§在新SOUL.md中是否存在
```

**修复方式**：将引用改为新§号，或将具体内容直接写入引用文件（消除跨文件引用）。

## 拆分后验证设计（2026-05-14 建立）

拆分完成后，执行"只读自检 + 动态验证设计"：

### 静态自检（5项）
1. **文件存在性**：所有预期文件是否存在、可读
2. **职责匹配**：每个文件的内容是否匹配其预期职责
3. **旧内容残留**：原文件中是否还有移出章节的详细内容
4. **引用完整性**：跨文件引用是否指向正确的新位置
5. **规则优先级**：SOUL.md > permission > risk > router > workflow > subagents > rules > templates

### 动态测试用例（按复杂度分级）
为每个 L0-L4 级别设计至少一个测试用例，验证：
- 路由是否正确（任务→复杂度→模型）
- Subagent 调用是否完整
- Risk Agent 是否在正确场景触发
- 确认/只读规则是否一致

**输出格式**：参考 `references/post-refactoring-audit-template.md`
