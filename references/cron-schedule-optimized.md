# Cron时间表优化（2026-05-28更新）

## 核心规则

1. **所有微信推送任务间隔≥10分钟**（5分钟是iLink临界值，不够安全）
2. **备份任务用`deliver: local`**，不走微信推送
3. **同一时段不允许多任务撞车**
4. **限流诊断**：`cronjob list` → `last_delivery_error` 字段

## 渠道级限流（2026-05-27发现）

微信iLink的限流是**渠道级**的——一旦触发，后面所有消息都被挡，等限流窗口恢复（通常1-2小时）才能正常。

**已知解决方案**：
1. **多渠道备份**：Telegram/Discord作为备用渠道（最稳）
2. **合并推送**：把多个报告合并成1条消息（最省事）
3. **本地缓存+手动拉取**：所有报告存本地，只推摘要（最不怕限流）

## 历史教训

- 2026-05-22：14个job中7个被限流，首次发现5分钟间隔不够
- 2026-05-27：15:35两个任务撞车→后续4个全部限流，主人从13:05后无推送
- 2026-05-27：从5分钟扩到10分钟间隔，解决撞车问题；渠道级限流需多渠道方案

## Gateway定时重启（2026-05-28更新）

**改为凌晨3点**。原cron每天触发6次（08:55/09:25/11:55/13:25/15:25/17:00），频繁重启打断会话。

**实现方式**：脚本内时间门控（`HOUR != "03"` → `exit 0`），cron仍触发但非3点时静默退出。原因：系统crontab无root权限修改。

**文件**：`~/.hermes/scripts/restart_gateway.sh`
